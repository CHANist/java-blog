When will Spring create bean, line 1 or line 2? 

public static void main(String[] args) {    AnnotationConfigApplicationContext context = new AnnotationConfigApplicationContext(AppConfig.class); // 1    UserService userService = context.getBean("UserService"); // 2   }

  • All non-lazy （非懒加载）singleton（单例） bean will be created in 1

  • Lazy （懒加载）beans or Prototype （原型，多例） Beans will be created in 2

@Service@Lazy // to specify it is a lazy beanpublic class UserService {    }

If not specified, by default, it is a non-lazy bean.

How to store data...

  1. Create a BeanDefinition type, which stores the BeanDefinition information, e.g. scope [Singleton / Prototype], type [what class type it is, for later creation], isLazy [whether the bean should be lazily initiated or not].
  2. Map<String, BeanDefinition> store beanName to beanDefinition, for easy retrieval of beanDefinition from beanName
  3. Map<String, Object> singletonObjects, which stores the mapping of beanName to the singleton objects.

public interface BeanPostProcessor {    default Object postProcessBeforeInitialization(Object bean, String beanName) {        return bean;    }        default Object portProcessAfterInitialization(Objcet bean, String beanName) throws BeanException {        return bean;     }}

As this Processor needs to be initiated before other beans. therefore it will be create in scan process, instead of normal createBean steps.

Why portProcessAfterInitialization need to return an object, what is the common use case of it? In Spring, it is used in Spring AOP.

@Componentpublic class SampleBeanPostProcessor implement BeanPostProcessor {    @Override    public Object portProcessAfterInitialization(Objcet bean, String beanName) throws BeanException {        if ("userService".equals(beanName)) {            Object proxy = Proxy.newProxyInstance(this.getClassLoader(), bean.getClass().getInterfaces(), new InvocationHandler() {                @Override                public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {                    System.out.println("切面逻辑");                    return method.invoke(bean, args);                }            });            return proxy;        }                return bean;     }}

For postProcessBeforeInitialization, it is commonly used to support custom annotation on some field.

Here is an example, 

@Retention(RetentionPolicy.RUNTIME)@Target(ElementType.FIELD)public @interface ZhouyuValue {    String value() default "";}
@Component("orderService")public class OrderServiceImpl implements OrderService {    @Autowired    private UserService userService;        @ZhouyuValue    private String test;
    public void test() {        System.out.println("test");    }}
@Componentpublic class SampleBeanPostProcessor implement BeanPostProcessor {    @Override    public Object postProcessBeforeInitialization(Objcet bean, String beanName) throws BeanException {        for (Field field: bean.getClass().getDeclaredFields()) {            if (field.isAnnotationPresent(ZhouyuValue.class)) {                field.set(bean, field.getAnnotation(ZhouyuValue.class).value());            }        }    }}

Spring also support getting the beanName from Spring, by implementing BeanNameAware interface.

有时候，你想知道你现在的BeanName 叫什么名字，那你可以implement 这个方法。

public interface BeanNameAware {    void setBeanName(String name);}

Of course, this is very easy to do. Indeed Spring support many different Aware, and we do not specifiy each of them one by one.

Indeed Spring does all the Aware in the same ApplicationContextAwareProcessor to call all **Aware interface, and is triggered in postProcessBeforeInitialization.
